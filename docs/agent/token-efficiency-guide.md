# AI 작업 요청 시 토큰 절약 가이드

이 문서는 AI 코딩 에이전트와 작업할 때 토큰 사용량을 줄이고, 답변 정확도를 높이기 위한 사용자용 가이드입니다.

## 왜 토큰이 많이 쓰이나요?

AI는 현재 대화 내용, 이전 요청, 읽은 파일, 도구 출력, 코드 내용을 모두 참고합니다. 그래서 아래 상황이 겹치면 토큰 사용량이 빠르게 늘어납니다.

- 처음에 아주 긴 프롬프트를 한 번에 입력한 경우
- 한 대화에서 기획, 구현, QA, 보안, 배포까지 계속 이어간 경우
- 큰 파일 하나에 기능이 많이 몰려 있는 경우
- `node_modules`, `.next`, 락파일 같은 불필요한 파일이 검색에 섞인 경우
- 룰 문서와 상세 명세가 루트에 많이 흩어져 있는 경우

## 가장 영향이 큰 부분

가장 큰 영향은 보통 긴 최초 프롬프트와 긴 대화 누적입니다. 한 번 입력한 긴 요구사항도 같은 대화 안에서는 계속 컨텍스트에 남아 후속 작업마다 비용을 만듭니다.

두 번째로 큰 영향은 큰 단일 파일입니다. 예를 들어 하나의 컴포넌트 파일에 대시보드, 관리자, 방 상세, 모든 Sheet와 폼이 들어 있으면 작은 버그를 고칠 때도 큰 파일 전체를 확인해야 합니다.

## 좋은 요청 방식

작업을 요청할 때는 아래처럼 범위를 좁혀 주세요.

```text
작업 목표:
- 방 삭제 오류 수정

읽어야 할 파일:
- src/app/actions/schedule-actions.ts
- src/components/app/ScheduleWorkspace.tsx
- supabase/migrations/*.sql

읽지 말아야 할 파일:
- node_modules
- .next
- package-lock.json
- design 이미지
- 전체 문서

수정 범위:
- 방 삭제와 관련된 서버 액션과 DB 함수만 수정

검증:
- npm run typecheck
- npm run lint
- npm run build
```

## 피하면 좋은 요청 방식

아래 방식은 토큰을 많이 사용하고, 작업 범위가 흐려질 수 있습니다.

```text
전체 프로젝트 다 확인해서 문제 있는 거 다 고쳐줘.
전에 말한 요구사항 전부 반영해서 전체적으로 다시 만들어줘.
디자인, DB, 보안, 배포, QA까지 한 번에 해줘.
```

이런 요청이 필요할 때도 있습니다. 다만 비용과 시간이 커지므로, 가능하면 단계별로 나누는 것이 좋습니다.

## 긴 프롬프트를 써야 할 때

초기 기획 단계에서는 긴 프롬프트가 도움이 됩니다. 대신 구현 단계로 넘어갈 때는 긴 프롬프트를 계속 대화에 들고 가지 말고 문서로 분리하는 것이 좋습니다.

추천 방식:

1. 긴 요구사항을 `docs/specs/product-spec.md`로 저장합니다.
2. AI에게 먼저 요약본을 만들게 합니다.
3. 실제 구현 요청에서는 요약본과 관련 파일만 읽게 합니다.

예시:

```text
docs/specs/product-spec.md는 전체 명세야.
이번 작업에서는 로그인 페이지만 구현해줘.
관련 규칙은 docs/agent/rules-summary.md만 읽어줘.
```

## 새 대화를 시작하면 좋은 시점

아래 상황에서는 새 대화를 시작하는 것이 좋습니다.

- 기획이 끝나고 구현으로 넘어갈 때
- UI 구현이 끝나고 QA로 넘어갈 때
- QA가 끝나고 보안 점검으로 넘어갈 때
- 배포 준비만 따로 하고 싶을 때
- 이전 대화가 너무 길어져서 답변이 느려질 때

새 대화를 시작할 때는 현재 상태 요약을 짧게 붙이면 됩니다.

```text
현재 프로젝트는 Next.js + Supabase 앱이야.
주요 기능은 구현됐고, 지금은 방 삭제 버그만 수정하려고 해.
관련 파일은 A, B, C야.
전체 명세는 읽지 말고 필요한 부분만 확인해줘.
```

## 프로젝트 문서 구조 추천

다음 구조를 추천합니다.

```text
AGENTS.md
docs/
  agent/
    rules-summary.md
    architecture.md
    tdd.md
    token-efficiency-rules.md
  specs/
    product-spec.md
    database-spec.md
  deployment/
    deployment.md
    security.md
```

루트에는 짧은 규칙 파일 하나만 두고, 상세 문서는 `docs` 아래로 분리하세요.

## 코드 구조에서 주의할 점

토큰을 줄이려면 코드도 작게 나누는 것이 좋습니다.

- 페이지 컴포넌트는 조립만 담당하게 합니다.
- 폼은 별도 컴포넌트로 분리합니다.
- Sheet와 Dialog는 별도 파일로 분리합니다.
- 서버 액션은 기능별 파일로 나눕니다.
- 1,000줄이 넘는 파일은 우선 분리 대상입니다.

## 매번 요청에 넣으면 좋은 문장

아래 문장을 요청 끝에 붙이면 토큰 낭비를 줄이는 데 도움이 됩니다.

```text
관련 파일만 읽고, node_modules, .next, package-lock.json, 전체 문서는 읽지 마.
필요하면 먼저 어떤 파일을 읽을지 짧게 말한 뒤 진행해줘.
```

## 기억할 핵심

- 긴 프롬프트는 한 번만 써도 같은 대화 안에서는 계속 비용이 됩니다.
- 큰 파일 하나는 작은 버그 수정도 비싸게 만듭니다.
- 루트 문서는 적을수록 좋고, 상세 문서는 주제별로 분리하는 것이 좋습니다.
- “전체 확인”보다 “이번 작업에 필요한 파일만 확인”이 훨씬 효율적입니다.
