# Token Efficiency Rules for AI Coding Projects

이 문서는 AI 코딩 에이전트가 프로젝트를 분석하고 수정할 때 불필요한 컨텍스트 사용을 줄이기 위한 일반 규칙입니다. 다음 프로젝트에서도 그대로 재사용할 수 있도록 특정 서비스나 도메인에 의존하지 않는 형태로 작성합니다.

## 1. 컨텍스트 범위 제한

- 작업 요청마다 이번 작업의 목표, 읽어야 할 파일, 읽지 말아야 할 파일을 명시한다.
- 에이전트는 작업과 직접 관련 없는 전체 프로젝트 파일을 먼저 읽지 않는다.
- 큰 변경이 아니라면 루트 문서, 전체 마이그레이션, 전체 타입 파일, 락파일을 자동으로 읽지 않는다.
- 특정 버그 수정은 관련 컴포넌트, 서버 액션, API, 데이터 접근 코드, 테스트 파일만 우선 확인한다.

## 2. 제외해야 할 파일과 폴더

AI 에이전트의 기본 검색 및 분석 범위에서 아래 항목을 제외한다.

- `node_modules`
- `.next`
- `dist`
- `build`
- `coverage`
- `.turbo`
- `.vercel`
- `.git`
- `package-lock.json`
- `pnpm-lock.yaml`
- `yarn.lock`
- 이미지, 영상, 바이너리 에셋
- `.env`
- `.env.local`
- `.env.*`
- 임시 폴더와 캐시 폴더

락파일은 의존성 충돌이나 설치 문제를 볼 때만 읽는다.

## 3. 긴 프롬프트 분리

- 하나의 프롬프트에 기획, UI, DB, 보안, 테스트, 배포 요구사항을 모두 넣지 않는다.
- 큰 프로젝트는 다음 단계로 나누어 요청한다.
  - 요구사항 정리
  - 데이터베이스 설계
  - UI 구현
  - 기능 구현
  - QA
  - 보안 점검
  - 배포 준비
- 이전 단계의 결과는 문서로 저장하고, 다음 작업에서는 필요한 문서만 참조한다.

## 4. 룰 문서 관리

- 루트에는 짧은 `AGENTS.md` 또는 `RULES.md` 하나만 둔다.
- 상세 규칙은 `docs/agent` 아래로 분리한다.
- 루트 규칙 파일은 80~120줄 이내를 권장한다.
- 상세 문서는 링크로 연결하고, 모든 작업에서 매번 전체 상세 문서를 읽게 하지 않는다.
- 오래된 프로젝트명, 예시 도메인, 사용하지 않는 기술 스택은 룰 문서에서 제거한다.

## 5. 큰 파일 분리

- 단일 소스 파일이 300줄을 넘기 시작하면 분리 후보로 본다.
- 700줄 이상이면 우선 분리 대상으로 본다.
- 1,000줄 이상이면 작업 정확도와 토큰 효율을 크게 떨어뜨리는 구조로 간주한다.
- UI, 상태 관리, 폼, Sheet/Dialog, 서버 호출, 유틸 함수를 한 파일에 몰아넣지 않는다.
- 페이지별 뷰, 공통 컴포넌트, 폼 컴포넌트, 상태 훅, 서버 액션을 분리한다.

## 6. 생성 파일과 원본 파일 구분

- 자동 생성 타입 파일은 필요한 경우에만 읽는다.
- 마이그레이션 SQL은 DB 작업이나 RLS 작업에서만 읽는다.
- 디자인 이미지와 Figma 자료는 UI 작업에서만 확인한다.
- 배포 문서는 배포 작업에서만 확인한다.
- 보안 문서는 보안 점검에서만 확인한다.

## 7. 문서 작성 규칙

- 긴 원본 명세는 `docs/specs`에 보관한다.
- AI가 매번 읽어야 하는 요약은 별도 `docs/agent/summary.md`로 짧게 유지한다.
- 기능별 결정사항은 한 문서에 계속 누적하지 말고 주제별로 나눈다.
- 완료된 일회성 작업 지시문은 운영 규칙 문서에 섞지 않는다.

## 8. 요청 템플릿

AI에게 작업을 요청할 때 아래 형식을 우선 사용한다.

```text
작업 목표:
- 

읽어야 할 파일:
- 

읽지 말아야 할 파일:
- node_modules
- .next
- package-lock.json
- design 이미지
- 전체 룰 문서

수정 범위:
- 

검증 방법:
- 
```

## 9. 에이전트 작업 원칙

- 먼저 `rg` 또는 제한된 `find`로 관련 파일만 찾는다.
- 파일 목록 명령에는 항상 제외 대상을 적용한다.
- 긴 파일은 전체를 읽기 전에 `rg -n`으로 관련 함수와 라인부터 찾는다.
- 대규모 리팩터링이 필요하지 않은 버그 수정에서는 관련 함수 주변만 읽는다.
- 최종 응답에는 읽은 주요 파일과 수정한 파일을 짧게 남긴다.

## 10. 새 프로젝트 시작 규칙

- 처음부터 모든 요구사항을 한 번에 구현시키지 않는다.
- 전체 명세는 문서로 저장하고, 첫 구현 요청은 MVP 범위만 지정한다.
- AI용 규칙 문서는 짧은 요약본과 상세본으로 분리한다.
- 프로젝트 초기에 `node_modules`, 빌드 산출물, 환경 변수 파일이 검색 범위와 Git에서 제외되도록 설정한다.
