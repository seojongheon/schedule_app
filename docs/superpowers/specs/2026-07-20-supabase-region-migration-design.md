# Supabase 리전 이전 설계

## 목적

기존 Supabase 프로젝트의 PostgreSQL 스키마, 업무 데이터, Auth 사용자와 비밀번호 해시를 새 리전의 Supabase 프로젝트로 이전한다. 이전 후 슈퍼관리자는 기존 로그인 ID와 비밀번호를 사용하되 새 프로젝트에서 새 세션을 발급받는다.

## 현재 조건

- 대상 애플리케이션은 Next.js, Supabase Auth, PostgreSQL, RLS를 사용한다.
- 실제 사용자는 이메일/비밀번호 방식의 슈퍼관리자 한 명이다.
- Supabase Storage에는 이전할 객체가 없다.
- 이전 시작 시점에 존재하는 데이터만 보존하면 된다. 이전 도중 발생하는 신규 변경의 동기화는 범위 밖이다.
- 기존 JWT 비밀키는 새 프로젝트에 복사하지 않는다. 기존 세션은 무효화하고 재로그인을 요구한다.
- 새 Supabase 프로젝트는 이미 생성되어 있으며, 복원 전 사용자 객체와 업무 데이터가 없는 상태여야 한다.

## 선택한 방식

Supabase CLI와 PostgreSQL `psql`을 사용한 전체 논리 백업·복원을 선택한다.

`supabase link`와 `supabase db push`만 실행하는 방식은 로컬 마이그레이션으로 스키마를 생성할 뿐 기존 `auth.users`, 비밀번호 해시, 프로필, 권한, 방, 일정 등의 행 데이터를 이전하지 않으므로 사용하지 않는다. 새 프로젝트에는 복원 전에 `db push`를 실행하지 않는다.

전체 논리 백업은 다음 세 부분으로 분리한다.

1. 데이터베이스 역할을 담은 `roles.sql`
2. 테이블, 함수, RLS, 트리거, 인덱스를 담은 `schema.sql`
3. Auth 및 업무 데이터를 담은 `data.sql`

새 프로젝트에는 역할, 스키마, 데이터 순서로 하나의 트랜잭션 안에서 복원한다. 데이터 복원 중에는 `session_replication_role = replica`를 사용해 트리거로 인한 중복 프로필 생성과 암호화 값의 이중 처리를 방지한다.

Supabase 공식 문서:

- 프로젝트 간 CLI 백업·복원: <https://supabase.com/docs/guides/platform/migrating-within-supabase/backup-restore>
- Auth 사용자와 비밀번호 해시 이전: <https://supabase.com/docs/guides/troubleshooting/migrating-auth-users-between-projects>
- 새 프로젝트 복원 범위와 수동 재설정 항목: <https://supabase.com/docs/guides/platform/clone-project>
- `db push` 동작 범위: <https://supabase.com/docs/reference/cli/supabase-migration-fetch>

## 전환 흐름

1. 기존 프로젝트와 새 프로젝트의 Project Ref, 리전, 연결 문자열을 각각 확인한다.
2. 기존 DB의 핵심 테이블별 행 수, 슈퍼관리자 UUID·이메일·프로필·권한, RLS와 주요 RPC 목록을 기준선으로 기록한다.
3. 기존 DB에서 `roles.sql`, `schema.sql`, `data.sql`과 마이그레이션 이력 백업을 생성한다.
4. 백업 파일이 비어 있지 않은지 확인하고 저장소 밖의 접근 제한된 임시 위치에 보관한다.
5. 새 프로젝트에서 필요한 비기본 PostgreSQL 확장 기능을 활성화한다.
6. 새 DB가 비어 있고 애플리케이션 트래픽을 받지 않는지 확인한다.
7. 새 DB에 역할, 스키마, 데이터를 원자적으로 복원한다.
8. 마이그레이션 이력을 복원해 이후 `supabase db push`가 이미 적용된 마이그레이션을 다시 실행하지 않도록 한다.
9. Auth 및 애플리케이션 환경 설정을 새 프로젝트에 맞춘다.
10. DB와 인증 무결성을 검증한 뒤 애플리케이션의 Supabase 환경 변수를 한 번에 교체하여 배포한다.
11. 슈퍼관리자가 기존 로그인 정보로 새 세션을 발급받고 핵심 기능을 검증한다.
12. 모든 승인 조건을 통과하면 새 프로젝트를 운영 대상으로 확정하고, 기존 프로젝트를 최소 7일 동안 변경 없이 보존한다.

## Auth와 환경 설정

DB 복원은 Auth 사용자 데이터는 옮기지만 Auth 설정과 API 키는 옮기지 않는다. 새 프로젝트에서 다음 항목을 수동으로 기존 정책과 맞춘다.

- 이메일/비밀번호 로그인 활성화
- Site URL과 허용 Redirect URL
- 회원가입 허용 여부와 이메일 확인 정책
- 비밀번호 정책, 세션 수명, 이메일 OTP 만료 설정
- SMTP 발신자와 이메일 템플릿(기존 프로젝트에서 사용한 경우)
- 소셜 로그인 비활성 상태

다음 환경 변수는 새 프로젝트에서 발급한 값으로 한 번에 교체한다.

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`
- `SUPABASE_SERVICE_ROLE_KEY` 또는 `SUPABASE_SECRET_KEY`
- DB 작업에서만 사용하는 `SUPABASE_DATABASE_PASSWORD`

다음 애플리케이션 비밀값은 기존 배포 값을 그대로 유지한다. 새로 생성하면 기존 암호화 데이터 또는 HMAC 기반 조회값을 읽거나 대조하지 못할 수 있다.

- `PRIVATE_DATA_ACTIVE_KEY_VERSION`
- `PRIVATE_DATA_KEY_V1`
- `SECURITY_HMAC_KEY`
- `DELETION_HMAC_KEY`
- 유지관리, 본인확인 및 공급자 연동에 사용하는 기타 서버 전용 비밀값

기존 프로젝트의 JWT 비밀키는 복사하지 않는다. 따라서 이전 프로젝트에서 발급한 액세스 토큰과 세션은 새 프로젝트에서 유효하지 않으며, 슈퍼관리자는 재로그인해야 한다. `auth.users`의 비밀번호 해시는 이전하므로 비밀번호 재설정은 요구하지 않는다.

## 검증 기준

### 데이터베이스

- 기존/신규 DB의 핵심 테이블별 행 수가 일치한다.
- `auth.users`의 슈퍼관리자 UUID와 이메일이 일치한다.
- 슈퍼관리자의 `profiles.id`가 `auth.users.id`와 일치한다.
- 슈퍼관리자에게 취소되지 않은 `super_admin` 역할 할당이 하나 이상 존재한다.
- 방, 일정, 참여자, 초대, 문의 및 감사 데이터에 외래키 고아 레코드가 없다.
- RLS 활성 상태, 정책, 주요 RPC, 트리거 및 인덱스가 존재한다.
- 마이그레이션 이력이 로컬 `supabase/migrations`의 버전과 일치한다.
- 기존 애플리케이션 암호화 키로 암호화된 값을 정상적으로 읽을 수 있다.

### 인증과 애플리케이션

- 새 publishable key를 사용하는 일반 클라이언트가 RLS를 우회하지 못한다.
- 새 secret 또는 service-role key를 사용하는 서버 작업이 정상 동작한다.
- 기존 로그인 ID와 비밀번호로 로그인하고 로그아웃할 수 있다.
- 새로 발급된 세션으로 슈퍼관리자 페이지에 접근할 수 있다.
- 일정 조회, 생성, 수정, 삭제와 방 조회가 정상 동작한다.
- 잘못된 비밀번호는 일반화된 오류로 거부된다.

## 위험과 통제

| 위험 | 통제 |
| --- | --- |
| 잘못된 프로젝트에 백업 또는 복원 | 실행 직전 기존/신규 Project Ref와 리전을 Dashboard와 CLI에서 교차 확인한다. |
| 비어 있지 않은 새 DB와 객체 충돌 | 복원 전에 대상 사용자 객체와 데이터가 없는지 검사하고 `db push`를 선행하지 않는다. |
| 트리거 실행으로 중복·이중 암호화 | 데이터 복원 구간에서 `session_replication_role = replica`를 사용한다. |
| Auth 사용자 또는 비밀번호 해시 누락 | `auth.users`의 UUID·이메일·암호화 비밀번호 존재와 실제 로그인을 각각 검증한다. |
| URL과 API 키가 서로 다른 프로젝트 값으로 혼합 | Supabase 관련 배포 환경 변수를 하나의 변경 단위로 교체하고 재배포한다. |
| 애플리케이션 암호화 키 손실 | 기존 서버 전용 키를 별도 비밀 관리 시스템에서 확인하고 그대로 이전한다. |
| 연결 문자열이나 백업의 유출 | 비밀값을 Git, 문서, 명령 인자 및 셸 기록에 남기지 않고 접근 제한된 임시 파일과 숨김 입력을 사용한다. |
| 마이그레이션 이력 불일치 | `supabase_migrations` 이력을 복원하고 전환 후 `supabase migration list`로 확인한다. |
| 새 DB에 기록 후 롤백하여 신규 데이터 유실 | 최종 승인 전 새 DB에 중요한 신규 데이터를 만들지 않는다. |

## 롤백

다음 중 하나라도 발생하면 새 프로젝트를 운영 승인하지 않는다.

- 슈퍼관리자가 기존 비밀번호로 로그인할 수 없다.
- 핵심 테이블 행 수, 사용자 UUID 또는 권한 할당이 다르다.
- RLS가 우회되거나 정상 사용을 차단한다.
- 기존 암호화 데이터를 읽을 수 없다.
- 일정과 방의 핵심 작업에서 오류가 발생한다.

애플리케이션 전환 후 문제가 발견되면 배포 환경 변수를 기존 Supabase URL과 키로 되돌리고 이전 배포를 재활성화한다. 승인 전 새 프로젝트에는 중요한 신규 데이터를 만들지 않으므로 역방향 데이터 동기화는 하지 않는다. 기존 프로젝트는 전환 후 최소 7일 동안 삭제, 비밀번호 변경, 스키마 변경 또는 데이터 정리를 하지 않고 롤백 원본으로 보존한다.

## 완료 조건

- 전체 백업과 복원이 오류 없이 완료된다.
- 데이터베이스 및 인증 검증 기준을 모두 통과한다.
- 슈퍼관리자가 기존 로그인 정보로 새 프로젝트에서 새 세션을 발급받는다.
- 애플리케이션 핵심 작업을 새 프로젝트에서 수행할 수 있다.
- 롤백용 기존 프로젝트와 백업의 보존 위치 및 폐기 예정일이 기록된다.

## 범위 밖

- Supabase Storage 객체 이전
- 소셜 로그인 공급자 이전
- 기존 세션 유지 또는 기존 JWT 비밀키 재사용
- 이전 도중 발생한 변경의 실시간 또는 증분 동기화
- 기존 프로젝트의 즉시 삭제
