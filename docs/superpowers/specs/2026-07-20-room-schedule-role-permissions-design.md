# 방 일정 소유권 및 역할별 권한 설계

## 목적

방 구성원이 자신의 일정을 직접 생성·수정·삭제할 수 있게 하고, 방장과 매니저는 방 구성원을 대신해 일정을 생성할 수 있게 한다. 일정 권한은 등록자가 아니라 명시적인 일정 소유자를 기준으로 판단한다. 방장과 매니저의 권한 차이, 구성원 강퇴 후 일정 보존, 초대 링크 생성 결함도 동일한 권한 모델 안에서 해결한다.

## 역할별 권한

| 역할 | 일정 생성 | 일정 수정 | 일정 삭제 | 초대 링크 | 구성원 강퇴·권한 변경 |
|---|---|---|---|---|---|
| 방장 | 방장·매니저·구성원을 소유자로 지정 | 본인 소유만 | 본인 소유 및 일반 구성원 소유 | 생성·재발급·취소 | 가능 |
| 매니저 | 방장·매니저·구성원을 소유자로 지정 | 본인 소유만 | 본인 소유 및 일반 구성원 소유 | 생성·재발급·취소 | 불가 |
| 구성원 | 본인 소유로만 생성 | 본인 소유만 | 본인 소유만 | 불가 | 불가 |
| 보기 전용 | 불가 | 불가 | 불가 | 불가 | 불가 |

추가 규칙은 다음과 같다.

- 방장이나 매니저가 다른 방장 또는 매니저 소유의 일정을 대신 생성할 수는 있지만, 생성 후에는 소유자 본인만 해당 일정을 수정하거나 삭제할 수 있다.
- 방장과 매니저는 일반 구성원 소유 일정을 삭제할 수 있지만, 자신이 소유하지 않은 방장·매니저 일정은 삭제할 수 없다.
- 보기 전용 참여자는 일정 소유자로 지정할 수 없고 일정 작성 권한도 없다.
- 초대 링크는 방장과 매니저만 관리한다.
- 구성원 강퇴, 매니저 지정·해제, 방장 위임은 방장만 수행한다.

## 일정 소유권 모델

`schedules`에 다음 필드를 추가한다.

- `owner_member_id`: 현재 일정 소유자인 활성 방 구성원. 수정·삭제 권한 판정에 사용한다.
- `owner_name_snapshot`: 일정에 표시할 소유자 이름. 소유권 이전이나 구성원 탈퇴·강퇴 후에도 기존 이름을 보존한다.
- `created_by_name_snapshot`: 최초 등록자 표시를 보존한다.

기존 `created_by_member_id`는 최초 등록자를 가리키는 감사 정보로 유지하되, 해당 구성원이 방에서 제거될 수 있도록 nullable 외래 키와 `on delete set null`로 전환한다. 기존 일정은 `owner_member_id = created_by_member_id`로 이전하고, 당시 구성원 닉네임으로 두 스냅샷을 채운다.

일정 소유자와 참여자는 서로 다른 개념이다. 소유자는 정확히 한 명이며 항상 참여자 목록에 포함한다. 방장·매니저는 일정 생성 시 소유자 외에 추가 참여자를 선택할 수 있다. 일반 구성원이 만드는 일정은 소유자와 참여자를 모두 본인으로 고정한다.

## 서버와 데이터베이스 권한

클라이언트에서 직접 여러 테이블을 순차 수정하지 않고 일정 생성·수정·삭제와 구성원 강퇴를 보안 정의 RPC로 처리한다. 각 RPC는 인증 사용자, 활성 계정, 현재 방 역할, 대상 일정 소유자의 역할을 데이터베이스에서 다시 조회한다.

### 일정 생성

- 일반 구성원은 자신의 `room_members.id`만 `owner_member_id`로 사용할 수 있다.
- 방장과 매니저는 같은 방의 `owner`, `manager`, `member`를 소유자로 지정할 수 있다.
- 보기 전용 사용자는 생성할 수 없고 소유자로도 지정할 수 없다.
- `created_by_member_id`는 항상 요청을 수행한 구성원으로 기록한다.
- 소유자를 `schedule_participants`에 자동 포함하고 추가 참여자가 모두 같은 방 소속인지 검증한다.

### 일정 수정

- 현재 인증 사용자가 `owner_member_id`의 사용자와 일치할 때만 수정할 수 있다.
- 일반 수정으로 `room_id`, `owner_member_id`, 소유자 이름 스냅샷, 최초 등록자 정보를 바꿀 수 없다.
- 참여자 변경 시에도 소유자는 참여자 목록에서 제거할 수 없다.

### 일정 삭제

- 일정 소유자는 자신의 일정을 삭제할 수 있다.
- 방장과 매니저는 현재 소유자 역할이 `member`인 일정만 추가로 삭제할 수 있다.
- 현재 소유자 역할이 `owner` 또는 `manager`인 일정은 소유자 본인 외에는 삭제할 수 없다.
- 보기 전용 사용자는 삭제할 수 없다.

### 구성원 강퇴

방장만 호출할 수 있는 하나의 트랜잭션에서 다음 작업을 수행한다.

1. 강퇴 대상이 같은 방의 일반 구성원 또는 매니저인지 확인한다. 방장 자신은 강퇴할 수 없다.
2. 대상이 소유한 일정의 `owner_member_id`를 현재 방장 구성원 ID로 변경한다.
3. `owner_name_snapshot`은 변경하지 않아 화면에 강퇴된 구성원의 기존 이름을 유지한다.
4. 대상이 최초 등록자인 일정은 외래 키의 `on delete set null`로 연결만 해제하고 `created_by_name_snapshot`을 유지한다.
5. 참여자 연결에서 강퇴 대상을 제거한다. 소유권이 방장에게 이전된 일정에는 방장을 참여자로 보장한다.
6. 방 구성원 행을 삭제한다.

이전된 일정의 실제 소유자는 방장이므로 방장만 수정·삭제할 수 있고 다른 매니저는 삭제할 수 없다. 화면에는 기존 소유자 이름과 최초 등록자 이름이 스냅샷으로 표시된다.

## UI 동작

- 새 일정 버튼을 방장·매니저·일반 구성원에게 표시하고 보기 전용 사용자에게는 숨긴다.
- 일반 구성원 폼에서는 소유자와 참여자를 본인으로 고정한다.
- 방장·매니저 폼에서는 방장·매니저·일반 구성원 중 일정 소유자를 선택하고 추가 참여자를 지정할 수 있다.
- 상세 화면의 수정 버튼은 일정 소유자에게만 표시한다.
- 삭제 버튼은 일정 소유자 또는 일반 구성원 소유 일정을 보고 있는 방장·매니저에게 표시한다.
- 참여자 권한 관리, 강퇴, 방장 위임 UI는 방장에게만 표시한다.
- 초대 링크 관리 UI는 방장과 매니저에게만 표시한다.
- 권한 거절 응답은 일반 오류로 숨기지 않고 사용자가 이해할 수 있는 안정적인 한국어 메시지와 403 상태를 반환한다.

## 초대 링크 생성 결함

현재 `record_invitation_outcome`은 성공한 `create`, `revoke`, `replace` 작업명을 `invitation_attempts.event_type`에 기록하지만 테이블 제약조건은 이 값을 허용하지 않는다. 권한 검사를 통과한 초대 생성 트랜잭션이 이 제약조건에서 롤백된다.

새 마이그레이션에서 기존 기록과 호환되도록 `invitation_attempts.event_type` 허용값을 `preview`, `validate`, `redeem`, `create`, `revoke`, `replace`, `deny`로 확장하고 TypeScript 데이터베이스 타입도 동기화한다. 초대 관리 RPC의 방장·매니저 권한은 유지한다.

## 마이그레이션과 호환성

- 새 additive 마이그레이션에서 소유권 필드, 스냅샷, 외래 키, 전용 RPC와 보호 트리거를 추가한다.
- 기존 일정의 등록자가 이미 삭제돼 매핑할 수 없는 행이 있다면 해당 방의 현재 방장에게 실제 소유권을 이전하고 기존에 남아 있는 표시 이름을 우선 사용한다.
- 데이터 백필과 제약조건 적용은 같은 마이그레이션에서 순서대로 수행해 nullable 중간 상태가 배포 후 남지 않게 한다.
- 애플리케이션 타입과 조회 컬럼을 새 필드에 맞춰 갱신한다.
- 기존 일정 참여자와 일정 내용은 삭제하지 않는다.

## 오류 처리와 원자성

- 일정 저장은 일정 본문과 참여자 변경을 하나의 트랜잭션으로 처리한다.
- 강퇴는 일정 이전, 참여자 정리, 구성원 삭제 중 하나라도 실패하면 전체를 롤백한다.
- 존재하지 않는 방·구성원·일정, 다른 방 구성원 지정, 비활성 계정, 권한 부족은 안정적인 오류 코드로 구분한다.
- 클라이언트는 실패한 요청을 성공한 것처럼 로컬 상태에 반영하지 않는다.

## 테스트

- 역할별 capability matrix가 정확한지 단위 테스트한다.
- 일반 구성원이 본인 소유 일정만 생성·수정·삭제할 수 있는지 검증한다.
- 방장·매니저가 다른 방장·매니저·구성원 소유 일정을 생성할 수 있는지 검증한다.
- 방장·매니저가 일반 구성원 일정은 삭제하지만 다른 방장·매니저 일정은 삭제하지 못하는지 검증한다.
- 매니저의 강퇴·등급 변경·방장 위임이 UI와 DB에서 모두 거절되는지 검증한다.
- 강퇴 시 일정이 방장에게 이전되고 기존 소유자 이름과 최초 등록자 이름이 유지되는지 검증한다.
- 일반 구성원 일정 폼이 본인을 소유자·참여자로 고정하는지 검증한다.
- 일정 소유자가 참여자 목록에서 제거되지 않는지 검증한다.
- 방장·매니저의 초대 생성·재발급·취소가 성공하고 일반 구성원은 거절되는지 검증한다.
- 전체 단위 테스트, 타입 검사, 린트, 프로덕션 빌드를 실행한다.

## 범위 제외

- 서비스 관리자 역할 변경
- 방 외부 사용자에게 직접 일정을 할당하는 기능
- 여러 명의 공동 일정 소유자
- 보기 전용 사용자에게 일정 작성 권한 부여
- 과거 강퇴 기록이나 별도 구성원 복구 화면
