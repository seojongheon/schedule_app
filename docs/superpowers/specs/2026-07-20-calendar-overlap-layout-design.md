# 일간 시간표 겹침 레이아웃 설계

## 목표

일간 시간표에서 일정 박스가 시간과 화면 양쪽에서 서로 침범하지 않게 배치한다. 같은 시간대에 겹치는 일정은 실제 등록 시각이 빠른 순서대로 왼쪽에서 오른쪽에 놓고, 새로운 일정이 추가되면 기존 박스도 같은 비율로 줄어든다. 같은 순간에 일정 5개 이상이 겹치는 구간에서만 일정 3개와 `+N 더보기` 한 칸을 표시한다.

## 현재 문제와 원인

현재 `ScheduleCalendar`는 실제 겹침 관계를 계산하지 않고 선택된 날짜의 배열 인덱스 홀짝으로 `right` 값을 정한다. 모든 박스에 동일한 `left`와 최소 너비도 적용되어 세 개 이상 겹치거나 부분적으로 겹치는 경우 박스가 서로 침범한다.

또한 압축 시간표는 24시간 캔버스 높이 자체를 420px로 줄이면서 일정 박스에는 픽셀 최소 높이를 유지한다. 짧거나 연속된 일정은 실제 시간 위치보다 크게 그려져 세로 방향에서도 겹칠 수 있다.

## 확정된 동작

### 겹침과 정렬

- 일정 구간은 `[시작, 종료)`로 본다. 한 일정의 종료와 다음 일정의 시작이 같으면 겹치지 않는다.
- 서로 겹치는 일정 사이에서는 `schedules.created_at`이 빠른 일정이 왼쪽, 늦은 일정이 오른쪽에 위치한다.
- 등록 시각이 같은 예외 상황에서는 일정 ID를 보조 정렬 키로 사용해 결과를 항상 동일하게 유지한다.
- 일정 시작·종료 경계로 하루를 시간 구간으로 나누고, 각 구간에서 실제로 활성 상태인 일정을 기준으로 열을 다시 계산한다.
- 활성 일정 수가 늘어나면 기존 박스도 같은 비율로 줄고, 줄어들면 남은 박스가 다시 넓어진다. 한 일정은 인접 구간의 배치가 같을 때 하나의 시각 조각으로 합친다.
- 겹치지 않는 일정은 시간표의 일정 영역 전체 너비를 사용한다.

### 다섯 개 이상 겹침

- 같은 순간의 활성 일정이 1~4개이면 모든 일정을 표시한다.
- 같은 순간의 활성 일정이 5개 이상이면 가장 먼저 등록된 일정 3개를 표시하고 네 번째 열에 `+N 더보기`를 표시한다.
- `N`은 해당 시간 구간에서 숨겨진 일정 수다.
- 시작과 종료가 이어져 하나의 연결 그룹을 이루더라도 어느 순간에도 5개가 동시에 겹치지 않으면 축약하지 않는다.
- 더보기 칸의 세로 범위는 실제로 5개 이상이 동시에 활성화된 시간 구간만 사용한다.
- 더보기 칸은 다른 표시 일정과 같은 비율의 네 번째 열을 사용하므로 박스를 덮지 않는다.

### 필터와 더보기 강조

- 기존 참여자 필터는 일정을 제거하지 않고 관련 없는 항목의 투명도를 낮추는 동작을 유지한다.
- 선택한 구성원의 일정이 숨겨진 목록에 하나라도 있으면 더보기 칸을 정상 강조 상태로 표시한다.
- 숨겨진 목록에 선택한 구성원의 일정이 없으면 더보기 칸도 다른 비관련 일정처럼 흐리게 표시한다.
- `내 일정`, `공동 일정`, 제목 검색에도 같은 `숨겨진 일정 중 하나라도 현재 조건과 일치` 규칙을 적용해 일관성을 유지한다.

### 더보기 목록

- 더보기 칸을 누르면 기존 `Sheet` 패턴의 목록을 연다.
- 목록에는 숨겨진 일정뿐 아니라 해당 혼잡 구간의 전체 일정을 표시한다.
- 목록 항목에는 일정 제목, 시작·종료 시각, 참여자 이름을 표시한다.
- 목록은 실제 등록 시각이 빠른 순서로 표시하고, 시간표에서 직접 보이는 일정과 숨겨진 일정을 구분하지 않는다.
- 목록에서 일정을 누르면 더보기 목록을 닫고 기존 `onScheduleClick` 경로로 일정 상세 화면을 연다.
- 키보드와 스크린 리더로도 사용할 수 있도록 더보기 칸과 목록 항목을 실제 버튼으로 구현하고 구체적인 접근성 이름을 제공한다.

## 구조와 데이터 흐름

### 일정 데이터

기존 DB의 `schedules.created_at`을 사용한다. 새 마이그레이션은 만들지 않는다.

- `Schedule` 엔티티에 `createdAt: string`을 추가한다.
- Supabase 일정 조회 컬럼에 `created_at`을 포함하고 `createdAt`으로 매핑한다.
- 목 데이터에도 고정된 등록 시각을 추가한다.
- DB 타입에는 이미 존재하는 `created_at` 계약을 그대로 사용한다.

### 순수 레이아웃 계산기

겹침 계산을 React 렌더링에서 분리한 순수 함수로 만든다. 입력은 일정 ID, 시작·종료, 등록 시각이고 출력은 일정 경계로 분할·병합된 다음 정보를 갖는다.

- 표시 일정 조각: `scheduleId`, 시작·종료, `columnIndex`, `columnCount`
- 축약 구간: `groupId`, 전체 일정 ID와 숨겨진 일정 ID 목록, 시작·종료, `columnIndex`, `columnCount`

함수는 모든 일정의 시작·종료 시각으로 하루를 겹치지 않는 구간으로 나눈다. 각 구간의 활성 일정을 등록 시각과 ID로 정렬한 뒤 네 개 이하이면 모두, 다섯 개 이상이면 앞의 세 일정과 더보기 칸을 출력한다. 인접한 출력 조각 중 일정 ID와 열 정보가 같으면 합쳐 불필요한 박스 분할을 줄인다. React는 이 결과를 백분율 `left`, `width`, `top`, `height`로 변환하고 일정 내용과 색상만 렌더링한다.

### 시간표 캔버스

스크롤 뷰포트와 24시간 캔버스를 분리한다.

- 일반 시간표와 압축 시간표 모두 내부 캔버스는 동일한 24시간 높이를 사용한다.
- `compact`는 보이는 스크롤 창 높이만 420px로 제한한다.
- 30분 일정이 한 줄 제목을 표시할 수 있는 높이를 확보하도록 짧은 일정 UI를 유지하되, 실제 30분 슬롯을 넘어서는 고정 최소 높이는 제거한다.

## 오류 및 경계 처리

- 잘못된 날짜나 종료가 시작보다 빠른 입력은 레이아웃 계산에서 제외하고 기존 데이터 검증 경계가 처리하도록 한다.
- 자정을 넘는 일정의 날짜 포함 기준은 현재 화면 계약을 유지하며 이번 변경에서 별도로 확장하지 않는다.
- 등록 시각이 누락된 레거시 클라이언트 데이터는 안정적인 마지막 순서로 배치하되, Supabase와 목 데이터 로더에서는 항상 값을 제공한다.
- 더보기 목록이 열린 상태에서 필터나 날짜가 바뀌어 그룹이 사라지면 목록을 닫는다.

## 테스트

순수 레이아웃 테스트에서 다음을 검증한다.

1. 겹치지 않는 일정은 모두 전체 너비 열을 사용한다.
2. 두 개와 세 개의 겹침은 각 시간 구간에서 등록 시각 순으로 왼쪽에서 오른쪽에 배치된다.
3. 늦게 등록된 일정이 시작되는 지점부터 기존 일정과 새 일정의 열 수가 함께 증가하고, 종료 후 남은 일정이 다시 넓어진다.
4. 종료와 시작이 같은 일정은 겹치지 않는다.
5. 같은 순간에 다섯 개 이상이면 가장 오래된 세 일정과 더보기 칸만 출력되고 숨김 수와 전체 목록이 정확하다.
6. 일정 다섯 개가 연쇄적으로 이어지지만 최대 동시 겹침이 네 개 이하이면 더보기 없이 모든 일정을 재배치한다.
7. 부분 겹침과 연쇄 겹침에서도 어떤 두 표시 박스도 같은 열과 같은 시간 구간을 공유하지 않는다.
8. 등록 시각 동률은 ID로 안정 정렬된다.

컴포넌트 계약 테스트에서 다음을 검증한다.

- 더보기 강조가 숨겨진 일정의 현재 필터 일치 여부를 사용한다.
- 더보기 목록에 해당 혼잡 구간의 전체 일정 제목과 참여자 이름이 표시된다.
- 목록 항목이 기존 일정 상세 콜백을 사용한다.
- 압축 시간표가 고정 높이 내부 캔버스를 사용한다.

## 범위 제외

- 주간 상단 요약과 월간 달력의 일정 표시 방식은 변경하지 않는다.
- 일정 저장 권한, 참여자 권한, 일정 데이터 자체는 변경하지 않는다.
- 드래그로 일정 시간을 바꾸는 기능이나 겹침 박스의 가로 스크롤은 추가하지 않는다.
